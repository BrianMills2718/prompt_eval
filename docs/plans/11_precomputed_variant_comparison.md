# Plan 11: Precomputed Variant Comparison

**Status:** 🚧 In Progress
**Type:** implementation
**Priority:** High
**Blocked By:** None
**Blocks:** grounded-research Tyler-literal versus calibrated-legacy evaluation

---

## Gap

**Current:** `prompt_eval` is strong when it owns the subject-model call, but
it does not yet provide a clean first-class path for evaluating frozen outputs
that were produced outside `prompt_eval` itself.

That is now the immediate cross-project need:

- `grounded-research` has one canonical Tyler-literal runtime
- older calibrated behavior should survive only as historical comparison, not a
  second runtime mode
- the comparison must happen over saved artifacts, not by reviving co-equal
  execution paths inside `grounded-research`

**Target:** add a small precomputed-output evaluation path that:

1. accepts variant outputs already produced elsewhere,
2. runs `prompt_eval` evaluators/judges over them,
3. returns a normal `EvalResult`,
4. works with `compare_variants()` and shared observability metadata.

**Why:** this keeps runtime code single-path while moving comparison into the
right layer.

## Pre-Made Decisions

1. This is an evaluation-only feature, not a second execution substrate.
2. The canonical output type remains `EvalResult`; do not create a separate
   incompatible result family.
3. The first slice will support precomputed per-input outputs keyed by
   `variant_name`, `input_id`, and optional `replicate`.
4. Per-trial evaluators and corpus evaluators must both work on precomputed
   outputs.
5. Observability should remain optional and should record that the outputs were
   precomputed rather than generated by `prompt_eval`.
6. The initial grounded-research use case is paired comparison of saved
   artifacts across the same frozen case set.

## Plan

### Step 1: Add typed precomputed-input contracts

- define a small typed surface for precomputed outputs
- require:
  - variant name
  - input ID
  - output payload
  - optional replicate
  - optional provenance metadata

### Step 2: Add a runner entrypoint for precomputed outputs

- evaluate the supplied outputs with the same evaluator interfaces used by
  `run_experiment()`
- produce a standard `EvalResult`
- preserve paired-by-input comparison compatibility

### Step 3: Add optional shared observability logging

- if enabled, log the evaluation family as precomputed-output evaluation
- do not fake subject-model execution metadata that never happened

### Step 4: Prove the grounded-research use case

- build a focused test fixture that compares two precomputed variants across a
  frozen shared case set
- verify:
  - `EvalResult` is valid,
  - `compare_variants(..., comparison_mode="paired_by_input")` works,
  - corpus evaluators still work,
  - optional observability metadata is truthful

## Failure Modes

| Failure Mode | Detection | Response |
|--------------|-----------|----------|
| The feature turns `prompt_eval` into a second execution substrate | API starts accepting subject-model call config instead of outputs | keep the input contract output-centric and reject runtime-execution creep |
| Precomputed evaluation creates a second result family | compare/query helpers need special branches | convert to standard `EvalResult` at the boundary |
| Observability lies about generation | logged runs imply subject-model execution occurred inside `prompt_eval` | log precomputed provenance explicitly or leave execution logging off |
| The grounded-research use case still needs bespoke code | cannot compare frozen Tyler-vs-legacy artifacts without local glue | widen the precomputed contract minimally, not by adding app-specific hooks |
